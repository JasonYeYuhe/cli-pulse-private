# v1.16 Ship Checklist — Companion CLI Helper

This document captures the FULL pipeline from "code complete" to "shipped to
end users". v1.16 introduces a Developer ID notarized .pkg distribution
channel for the Python helper alongside the existing MAS app, so this
checklist has more steps than a typical release.

**Branch state**: `v1.16-impl`. Plan committed on `v1.16-plan`. Both ready for
user testing / sign-off / merge.

---

## Phase 1 — One-time infrastructure (user action required)

These are setup steps the user performs once. After this is done, every
release is automated.

### 1.1 — Developer ID Installer certificate

The MAS app distribution uses Apple Distribution certs. The .pkg helper
distribution requires a separate **Developer ID Installer** cert.

1. Visit https://developer.apple.com/account/resources/certificates/list
2. Click `+` → "Developer ID Installer"
3. Generate CSR via Keychain Access (Certificate Assistant → Request a
   Certificate from a Certificate Authority)
4. Upload CSR, download generated cert
5. Double-click to install in login keychain
6. Verify: `security find-identity -v -p basic | grep "Developer ID Installer"`
   should show: `Developer ID Installer: Yuhe Ye (KHMK6Q3L3K)`

We already have the **Developer ID Application** cert (used for embedded
Mach-O signing). Confirm it's still in keychain:

```
security find-identity -v -p codesigning | grep "Developer ID Application"
```

### 1.2 — Notarization profile

```sh
xcrun notarytool store-credentials AC_NOTARY_PROFILE \
    --apple-id "your-apple-id@example.com" \
    --team-id KHMK6Q3L3K \
    --password "<app-specific-password-from-appleid.apple.com>"
```

App-specific password is generated at https://account.apple.com → Security →
App-Specific Passwords (NOT your real Apple ID password). Profile name
`AC_NOTARY_PROFILE` matches what `scripts/build_helper_pkg.sh` looks for.

### 1.3 — Public mirror repo for helper releases

The MAS app downloads `cli-pulse-helper-<ver>-arm64.pkg` from a public GitHub
Releases endpoint. Create a public repo for this:

1. Create `JasonYeYuhe/cli-pulse-helper-releases` (public, empty)
2. Each helper release uploads:
   - `cli-pulse-helper-1.16.0-arm64.pkg` (the notarized + stapled pkg)
   - `cli-pulse-helper-1.16.0-arm64.pkg.sha256`
3. The "latest" tag points to the most recent release; the manifest URL the
   MAS app fetches is:
   ```
   https://github.com/JasonYeYuhe/cli-pulse-helper-releases/releases/download/latest/latest.json
   ```
4. `latest.json` content (uploaded as a release asset on the `latest` tag):
   ```json
   {
     "version": "1.16.0",
     "arch": "arm64",
     "url": "https://github.com/JasonYeYuhe/cli-pulse-helper-releases/releases/download/v1.16.0/cli-pulse-helper-1.16.0-arm64.pkg",
     "sha256": "<from build_helper_pkg.sh output>",
     "size_bytes": <bytes>,
     "min_os_version": "13.0",
     "release_notes_url": "https://github.com/JasonYeYuhe/cli-pulse-helper-releases/releases/tag/v1.16.0"
   }
   ```

The HelperInstaller.swift in CLIPulseCore reads this manifest, fetches the
.pkg, verifies the SHA-256, and hands it to `NSWorkspace.open`.

## Phase 2 — Per-release build (automatable later)

Each helper release goes through this sequence. Run from the repo root.

### 2.1 — Build the helper .pkg

```sh
DEV_ID_APP="Developer ID Application: Yuhe Ye (KHMK6Q3L3K)" \
DEV_ID_INSTALLER="Developer ID Installer: Yuhe Ye (KHMK6Q3L3K)" \
NOTARY_PROFILE="AC_NOTARY_PROFILE" \
scripts/build_helper_pkg.sh
```

Output:
```
build/v1.16-pkg/cli-pulse-helper-1.16.0-arm64.pkg     (signed + notarized + stapled)
build/v1.16-pkg/cli-pulse-helper-1.16.0-arm64.pkg.sha256
build/v1.16-pkg/manifest-fragment-arm64.json
```

The script also produces the embedded `CLI Pulse Helper Uninstaller.app`
(via `scripts/build_helper_uninstaller.sh`, invoked transitively).

### 2.2 — Smoke test on a clean VM

PER `feedback_v080_crash_on_launch_incident.md`: **never** promote to
"latest" without VM smoke. The `~$0.50/hr` cost of an Azure macOS VM is
worth it.

On a clean macOS 15 VM (no developer tools, no prior CLI Pulse install):

1. Install MAS app from local archive (or wait until v1.16 is in the App
   Store and install from there)
2. Pair the app with the Supabase account
3. Open Pairing tab in Settings → click "Install Companion CLI"
4. Verify: download progress → Gatekeeper "downloaded from internet" prompt
   → Installer.app opens → Continue + Install (no admin password) → 3-5
   seconds → installer closes → app shows "v1.16.0 running ✓"
5. Try spawning a Codex / Gemini / Claude session from the picker

Spot-check:
```sh
launchctl list | grep cli-pulse                # should show running PID
ls ~/Library/CLI-Pulse-Helper/                 # should have cli_pulse_helper, _internal/, Uninstaller.app
spctl --assess --type install --verbose cli-pulse-helper-1.16.0-arm64.pkg
```

### 2.3 — Migration test from v1.15

On a test machine with a manually-running v1.15 `nohup python3
helper/cli_pulse_helper.py daemon`:

1. Verify v1.15 helper is alive: `lsof -t ~/Library/Group\ Containers/group.yyh.CLI-Pulse/clipulse-helper.sock`
2. Click "Install Companion CLI" in v1.16 app
3. Watch postinstall log: `tail -f ~/Library/Logs/CLI-Pulse-Helper/postinstall.log`
4. Verify: postinstall.sh detects + SIGTERMs the v1.15 helper, waits for
   graceful drain, then bootstraps the v1.16 LaunchAgent
5. Check both old + new helpers: only the new one should be running

### 2.4 — Update test (v1.16.0 → v1.16.1, future releases)

Bump HELPER_VERSION in `helper/system_collector.py`, rebuild pkg, upload to
mirror repo, click "Update Companion CLI" in the app. Verify: re-runs install
flow, postinstall stops old + bootstraps new, app reports new version.

### 2.5 — Uninstall test

In the app: click "Uninstall…" → Helper Uninstaller.app opens → confirm →
"Uninstalled" alert → quit.

Verify cleanup:
```sh
launchctl list | grep cli-pulse                # empty
ls ~/Library/LaunchAgents/yyh.cli-pulse.helper.plist  # gone
ls ~/Library/CLI-Pulse-Helper/                 # gone (trampoline cleaned it)
```

## Phase 3 — MAS app submission

### 3.1 — Build + upload v1.16.0 (build 55)

The MAS app itself ships unchanged distribution-wise. Use the existing
flow:

```sh
scripts/build-appstore.sh build_macos
```

This produces an MAS-signed archive WITHOUT the Phase 4D LaunchSwift helper
(stripped, per `feedback_mas_vs_devid_helper.md`). The CLIPulseHelper
LoginItem (sandboxed, data-sync only) ships as before.

Upload via Transporter or `scripts/build-appstore.sh upload`.

### 3.2 — App Store Connect description

For v1.16's "What's New" section, use this framing (per Gemini's
Q2 review of plan §1.11):

> Multi-CLI managed sessions: from the Pairing tab, install our optional
> Companion CLI Helper (a Developer ID notarized download from
> github.com/JasonYeYuhe/cli-pulse-helper-releases) to drive `claude`,
> `codex`, and `gemini` CLI sessions directly from the menubar. The
> helper is signed by Apple's notary service and installs to your home
> folder with no admin password required. Optional and opt-in.

The "optional companion helper" framing aligns with App Store Review
Guideline 2.5.2 (1Password 7 precedent).

### 3.3 — Submit for review

Standard ASC submission flow. Most likely accepted on first try since the
managed-CLI feature stays out of the MAS bundle (helper is downloaded
post-install from the user-action button).

If rejected for the "downloads code" pattern: respond with the 1Password
precedent + explain the helper is opt-in, signed, notarized.

- Watch the Sessions tab for regressions in §2.x fixes (Codex 101 →
  should now show panic message; Gemini 401 → at most one log line per
