# v1.19 — Developer ID DMG distribution channel

A second distribution channel for CLI Pulse Mac running parallel to
the existing Mac App Store (MAS) path. Direct-download `.dmg` hosted
on a public GitHub repo, signed with Developer ID Application,
notarized + stapled, consumed in-app via `AppUpdater` (manifest +
SHA-256 + version compare + URLSession download).

## Why this channel exists

- Three pre-built branches (B3 helper-label-collision, B3-bis
  helper-config-bridge, v1.18.2-impl) are **MAS-strip-inert**:
  `build-appstore.sh` strips the unsandboxed `cli_pulse_helper`
  LaunchAgent before MAS upload, so the fixes never run in
  production. The DMG channel is the ship vehicle.
- ASC review queue (1-5 day median) is the bottleneck for iterating
  on the macOS app. DMG channel lets us push fixes in ~30 minutes
  from commit.
- Power-user / dev-community beta tier becomes viable.

## What ships in this channel

- The full `.app` bundle (same source tree as MAS), signed with
  **Developer ID Application** instead of Apple Distribution.
- The Swift `cli_pulse_helper` LaunchAgent at
  `Contents/Helpers/cli_pulse_helper` (stripped from MAS builds —
  see [feedback_mas_vs_devid_helper](../docs/feedback_mas_vs_devid_helper.md)).
- The same Python helper `.pkg` from
  [cli-pulse-helper-releases](https://github.com/JasonYeYuhe/cli-pulse-helper-releases)
  drives managed CLI sessions (no channel-specific helper).

## What does NOT ship in this channel

- v1.19.0 ships **arm64-only**. Universal binary in v1.19.x if user
  demand surfaces.
- v1.19.0 ships the existing Python helper for managed CLI —
  Phase 4D/4E Swift LaunchAgent runtime cutover is deferred to
  v1.19.x or v1.20.
- No StoreKit cloud-backend bypass yet — DEVID users get
  premium tier locally; server-validated premium endpoints will
  reject DEVID requests until v1.19.1 backend allow-list lands
  (see [Subscription tier on the DEVID channel](#subscription-tier-on-the-devid-channel)).
- No CI publishing — v1.19.0 release publish is manual.

## Build pipeline

Two scripts, top-level `scripts/`:

### `scripts/build_signed_app.sh` (extended in v1.19)

Foundation script. Builds the helper + macOS app + embeds + bottom-up
codesign with entitlements preservation. Two new opt-in env knobs:

| Env var | Purpose | Default |
|---|---|---|
| `CODESIGN_TIMESTAMP` | When set (any value), pass `--timestamp` to codesign for Apple's timestamp service (required for notarization). Otherwise `--timestamp=none` (current dev/CI behaviour). | unset |
| `DEVID_BUILD_FLAG` | When set, append `SWIFT_ACTIVE_COMPILATION_CONDITIONS=$(inherited) DEVID_BUILD` to the xcodebuild call. Enables `#if DEVID_BUILD` code paths in Swift. | unset |

Backward-compatible: CI's `./scripts/build_signed_app.sh Debug` continues to work
exactly as before.

### `scripts/build_devid_dmg.sh` (new in v1.19)

Orchestrates the full Developer ID DMG pipeline:

1. Invokes `build_signed_app.sh Release` with `CODE_SIGN_IDENTITY="$DEV_ID_APP"`,
   `CODESIGN_TIMESTAMP=1`, `DEVID_BUILD_FLAG=1`.
2. Notarizes the `.app` via `xcrun notarytool submit --wait`.
3. Staples the `.app`.
4. Packages as `.dmg` via `hdiutil create -volname "CLI Pulse" -format UDZO -fs HFS+`.
5. Signs the DMG with Developer ID Application + timestamp.
6. Notarizes the DMG, staples it.
7. Verifies via `spctl --assess`.
8. Emits `<dmg>.sha256` + `manifest-fragment-arm64.json` for the
   public mirror repo.

#### Required env

```sh
DEV_ID_APP="Developer ID Application: Yuhe Ye (KHMK6Q3L3K)"
```

#### Notarytool credentials (G8 — pick one mode)

**Local mode** (default — uses keychain profile from
[reference_devid_installer_cert](../docs/reference_devid_installer_cert.md)):
```sh
NOTARY_PROFILE=AC_NOTARY_PROFILE   # default if unset
```

**Inline mode** (CI-friendly — overrides keychain profile when all
three vars are set):
```sh
APPLE_NOTARY_USER="yyyyy.yeyuhe@icloud.com"
APPLE_NOTARY_APP_PASSWORD="xxxx-xxxx-xxxx-xxxx"
APPLE_TEAM_ID="KHMK6Q3L3K"
```

If the keychain profile vanishes again (per
[feedback_keychain_notary_vanished](../docs/feedback_keychain_notary_vanished.md)),
the persistent password backup at
`~/Library/Application Support/CLI-Pulse-Secrets/notarytool-app-password-2026-05-12.txt`
contains the password to re-run `xcrun notarytool store-credentials`.

#### Skip flags

- `--skip-sign` — ad-hoc sign only; auto-implies `--skip-notarize`. Use for structural smoke.
- `--skip-notarize` — sign but don't notarize. Local diagnostic builds.
- `--dry-run` — echo commands but don't execute.

#### Output

```
build/v1.19-dmg/
├── staging/CLI Pulse.app             # signed, notarized, stapled
├── CLI-Pulse-1.19.0-arm64.dmg        # signed, notarized, stapled
├── CLI-Pulse-1.19.0-arm64.dmg.sha256
└── manifest-fragment-arm64.json
```

## App-side updater

### `AppUpdater.swift` (new in v1.19)

Lives in `CLIPulseCore/Sources/CLIPulseCore/`. Mirrors the structure
of `HelperInstaller.swift` (manifest fetch + SHA-256 verify + version
compare + URLSession download). Key differences:

- Manifest URL points to
  `https://github.com/JasonYeYuhe/cli-pulse-distrib/releases/download/latest/latest.json`
- Manifest schema adds `build` + `channel` fields for future routing
- Cache-policy override on manifest fetch (G7 — avoids stale
  `latest.json` on the client side)
- `install()` opens the DMG in Finder + terminates self 500ms later
  (G2 — Finder can't drag-replace a running `.app`)
- Gated via `#if DEVID_BUILD`; MAS builds never construct an updater

### `AppPermissionMigrationChecker.swift` (new in v1.19)

G1 mitigation. macOS TCC binds Notifications / Accessibility / etc.
to the app's Designated Requirement (cert chain + bundle ID + team
ID). Switching from Apple Distribution (MAS) to Developer ID
Application (DEVID) changes the DR, so previously-granted permissions
silently revert.

The checker writes a snapshot of granted permissions to app-group
UserDefaults from **both** MAS and DEVID builds (same call path).
DEVID builds also compare against the previous snapshot and surface
a banner in `AppUpdaterSection` if permissions reverted, with
deep-links to System Settings panes.

Read-only; does NOT call `requestAuthorization` (that remains owned
by `DataRefreshManager`).

### `AppUpdaterSection.swift` (new in v1.19)

SwiftUI section in Settings. Three banners:

1. **G1 — permission migration nudge** (conditional): "Migrating from
   App Store version reset permissions. Re-grant Notifications /
   Accessibility?" with deep-links + dismiss button.
2. **G5 — MAS auto-update warning** (permanent): "You're on the Beta
   channel. Disable App Store auto-updates so the App Store version
   doesn't silently overwrite this beta."
3. **State machine UI**: checking / upToDate / updateAvailable /
   downloading / readyToInstall / error — mirrors `CompanionCLISection`.

Gated via `#if DEVID_BUILD` so MAS builds compile out the entire
file.

## Bundle identifier strategy (SR2 decision)

Same `yyh.CLI-Pulse` bundle ID for both channels. User installs MAS
OR DEVID, not both. Migration = drag-replace from the new DMG.

Same app-group `group.yyh.CLI-Pulse` for both channels. Pairing
state, bookmarks, helper config flow naturally across channel
switches.

## Subscription tier on the DEVID channel

v1.19.0 SR1 (commit `a03962c`, landed 2026-05-12):
`SubscriptionManager.updateCurrentEntitlements()` short-circuits at
the top under `#if DEVID_BUILD`, setting `currentTier = .pro` +
`isLifetime = true` + `tierResolutionState = .resolvedConfirmed` +
`lastTierRefreshSource = "devid-beta-channel"`. The previous "not
yet implemented" note in this doc was pre-SR1. Note the real
`SubscriptionTier` enum case is `.pro` (no `.premium` case exists).

v1.19.1+: server-side allow-list bypass via `X-CLI-Pulse-Channel: beta`
HTTP header. Backend treats beta channel as "skip receipt validation."
Requires a backend schema touch — user authorization needed per
autonomy contract `feedback_cli_pulse_autonomy.md` §"When to flag" #1.

## Releasing v1.19.x

These steps are NOT in v1.19 scope yet — they require user
authorization for public-repo writes per autonomy contract.

1. Bump `CLI Pulse Bar/CLI Pulse Bar/Info.plist`
   `CFBundleShortVersionString` to `1.19.0` (and CFBundleVersion).
2. Run `scripts/build_devid_dmg.sh` with DEV_ID_APP set.
3. Clean-Mac smoke (MANDATORY per
   [feedback_v080_crash_on_launch_incident](../docs/feedback_v080_crash_on_launch_incident.md))
   — install the DMG on a fresh user account / VM, confirm app
   launches + no Gatekeeper prompt + sandbox + entitlements intact.
4. User authorizes new public repo `cli-pulse-distrib`:
   ```sh
   gh repo create JasonYeYuhe/cli-pulse-distrib --public \
       --description "CLI Pulse Mac — Developer ID DMG releases"
   ```
   Seed with README (empty repo can't host releases — see
   [feedback_v116_helper_pkg_shipped](../docs/feedback_v116_helper_pkg_shipped.md) §8).
5. Publish:
   ```sh
   gh release create app-v1.19.0 \
       --repo JasonYeYuhe/cli-pulse-distrib \
       --title "CLI Pulse Mac v1.19.0 (Developer ID Beta)" \
       --notes-file release-notes.md \
       build/v1.19-dmg/CLI-Pulse-1.19.0-arm64.dmg \
       build/v1.19-dmg/CLI-Pulse-1.19.0-arm64.dmg.sha256
   ```
6. Publish `latest.json` (delete + recreate prerelease tag).
   **The asset MUST be named `latest.json`** — `AppUpdater` fetches
   `releases/download/latest/latest.json`, so uploading the file
   as its build-time name `manifest-fragment-arm64.json` produces a
   404 on every client update check. Use `gh`'s `path#name` syntax
   to rename the asset at upload time:
   ```sh
   gh release delete latest --repo JasonYeYuhe/cli-pulse-distrib --yes
   gh release create latest \
       --repo JasonYeYuhe/cli-pulse-distrib \
       --title "Latest app manifest" \
       --notes "Currently points to v1.19.0." \
       --prerelease \
       'build/v1.19-dmg/manifest-fragment-arm64.json#latest.json'
   ```
   (Single-quote the asset spec so the shell doesn't interpret `#`
   as a comment marker — bash and zsh both do.)
7. Verify: `curl -sL https://github.com/JasonYeYuhe/cli-pulse-distrib/releases/download/latest/latest.json | jq .version`
   → must return `"1.19.0"`. A 404 here means the asset name didn't
   become `latest.json`; redo step 6 with the `#latest.json` rename.

## Rollback

If the v1.19.x release crashes on a user's Mac:

1. `gh release delete latest --yes` + recreate pointing at the
   previous good version's `latest.json` (or skip if no prior
   v1.19.x exists).
2. `gh release edit app-v1.19.0 --notes "⚠️ YANKED — see release notes"`.
3. Users who already installed are unaffected by latest.json rollback
   (they have the broken .app). Communicate via the GitHub release
   page + (if available) Sentry-driven alert in the app.

This rollback pattern matches the [v0.8.0
incident](../docs/feedback_v080_crash_on_launch_incident.md).
